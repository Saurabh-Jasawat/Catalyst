1. Introduction & Product Workflow

Catalyst is an intelligent MERN (MongoDB, Express, React, Node.js) study planner, syllabus tracking engine, and exam preparation companion. Designed for high-stakes competitive examinations (such as UPSC CSE, IIT JEE, NEET, SSC CGL, and GATE), Catalyst empowers users to visualize, track, and optimize their preparation journey through spaced repetition, custom study logs, mock exam analysis, past-year papers analysis, and a structured syllabus database.

How the Application Flows:

  1. Onboarding & Authentication: Users sign up using their full name, email, and password. Access is granted instantly, generating a secure JSON Web Token (JWT) in Node.js and saving their account details to MongoDB.
  2. Exam Selection Workspace: Users select one or more competitive exam tracks (templates like UPSC, JEE, custom tracks, etc.). The system configures the workspace, instantiating their syllabus checklist.
  3. Syllabus Mastery Tracker: Users break down subjects into topics and subtopics. As subtopics are updated (from Theory to Practice, Revision, Mock, and Completion), the system automatically updates progress circles, calculates overall readiness percentages, and increments XP (experience points) and streaks.
  4. Database Sync: Any updates on syllabus completion, notes edits, revision dates, or planners are captured by a 1-second debounce synchronization timer that saves the details to MongoDB Atlas automatically in the background.
  5. Password Reset with OTP: If a user forgets their password, they trigger an OTP dispatch. The server generates a temporary, cryptographically safe, 15-minute 6-digit verification code, sends it to the user's Gmail, and expects them to type it back along with their new password.

Authentication Middleware (backend/middleware/auth.js)

Purpose: Validates JWT tokens in the request authorization header.

const jwt = require('jsonwebtoken');

module.exports = function (req, res, next) {
  const authHeader = req.header('Authorization');
  
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({ msg: 'No token, authorization denied' });
  }

  try {
    const token = authHeader.split(' ')[1];
    const decoded = jwt.verify(token, process.env.JWT_SECRET || 'catalyst_super_secret_jwt_key_123456');
    
    // Decoded token contains user id, e.g. { id: "user_id" }
    req.user = decoded;
    next();
  } catch (err) {
    res.status(401).json({ msg: 'Token is not valid' });
  }
};

Why does Node.js use a single-threaded architecture? How does it handle concurrency differently from multi-threaded platforms?
Traditional servers (like Apache) spawn a separate thread for every incoming client connection. This consumes significant memory and CPU resources during high concurrency states. Node.js uses a single main thread for handling JavaScript execution, paired with an asynchronous Event Loop (powered by the C++ library libuv). When an I/O operation (like reading a MongoDB collection or sending an email) starts, Node delegates the operation to the operating system's system threads or internal thread pools, instantly freeing up the main event loop thread to accept other connections. This design is highly concurrent and lightweight, particularly for I/O-intensive web applications.
Explain the difference between SQL and NoSQL. Why was MongoDB chosen for Catalyst?
SQL databases are relational, using table structures with fixed schemas and foreign-key constraints. NoSQL databases (like MongoDB) are document-oriented, storing data in flexible JSON-like documents. MongoDB was selected for Catalyst because study trackers deal with rich, dynamic syllabus profiles that can contain deeply nested lists (subjects -> topics -> subtopics -> subtopic details). Storing this in SQL would require complex multi-table joins, while MongoDB can store the entire exam profile in a single document, optimized for fast reads and writes.
How does JWT verify tokens? What is in the Header, Payload, and Signature?
A JSON Web Token (JWT) has three sections:
  • Header: Specifies the cryptographic algorithm (e.g. HS256) and type.
  • Payload: Contains user-specific claims (e.g., userId) and expiration times.
  • Signature: Generated by signing the encoded header, encoded payload, and a secret key (JWT_SECRET).
Verification works by having the server take the header and payload from the client's request, re-sign them with its server-side secret key, and check if the result matches the client's signature. If they match, the token is verified.
