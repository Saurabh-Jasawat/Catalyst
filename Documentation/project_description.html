Project Description Document

Technical Details, Modules, Database Schema, and Flow Architecture

1. Detailed Module Descriptions

1.1 User Authentication & Security Module

Protects access to study workspaces. Supports signup, login, JWT token generation, auth route guards, and email OTP verification. When requesting a password reset, a 6-digit code is dispatched to the user's Gmail using Nodemailer SMTP. Once validated, the backend cryptographically updates the password using bcrypt.

1.2 Syllabus Tracking Module

Maps subjects, topics, and subtopics into a progress matrix. Users can toggle status across five preparation stages: Foundation, Practice, Revision, Mock Test, and Completed. Each stage update recalculates overall percentages and awards XP.

1.3 Spaced Repetition Revision Module

Calculates next revision dates automatically based on intervals (e.g. 1 day, 3 days, 7 days, 30 days) once a user logs a review session. It surfaces overdue topics directly on the Spaced Repetition queue.

1.4 PYQ Analysis & Notes Module

Logs past year question paper scores, attempt status, and error logs. Users can directly record general notes, formula lists, and common traps (mistakes register) for subtopics, linking syllabus theory to practice.

1.5 Dashboard Metrics & Planner Module

Features overall statistics widgets, interactive countdowns (D-Day phases), target scores, daily task checklists, and a study hours chart showing logs across past days.

2. Technical Rationale & Design Choices

  • React Context API: Used for global state management to prevent prop drilling and keep active exam details, authentication data, and user profiles synchronized.
  • Keystroke Debouncing (1s): Throttles HTTP PUT requests during active typing, reducing write operations on MongoDB to secure connection pools.
  • Mongoose Models & Schema: Models user, exam, and backup profiles, enforcing structured types and relational index constraints (such as compound unique index on userId and examName).

3. Database Design (MongoDB Collection Schemas)

3.1 User Schema (users)

Field Name Data Type Constraint / Description
name String Required, trimmed name
email String Required, unique index, lowercase
password String Required, bcrypt hash
enrolledExams Array of Strings Default: ['UPSC CSE']
resetCode String Default: null (used for password reset OTP)
resetCodeExpires Date Default: null (OTP expiry timestamp)

3.2 ExamData Schema (examdatas)

Field Name Data Type Constraint / Description
userId ObjectId Ref: User model (foreign relation)
examName String Required, matches active track
streak Number Default: 1
points (XP) Number Default: 250
syllabusData Array of Subjects Enforces nested topic and subtopic arrays
pyqPapers Array of Papers Stores custom and template papers list

4. System Data Flow Pipeline

  1. User Mutation Event: User completes a topic or adds a note. React state variables are immediately updated in AppContext.
  2. Debounce Window: The system waits for the user to stop editing for 1 second.
  3. Network Request: The client triggers HTTP PUT request to /api/exam/:examName including the updated state payload in the JSON body.
  4. Authentication Validation: Express router invokes JWT auth middleware to verify authorization headers.
  5. Database Transaction: Mongoose processes a findOneAndUpdate query with upsert enabled, writing the document to MongoDB.
